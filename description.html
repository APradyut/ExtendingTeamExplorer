Introduction

Team Explorer in Visual Studio 2012 offers multiple options for extending the user experience.  This sample demonstrates several of the extensibility points by:
  • Adding a new "Recent Changes" Team Explorer page with two sections.
  • Adding a new "Selected File Info" section to the built-in Pending Changes page.
  • Adding a "Recent Changes" navigation link under the Pending Changes navigation item.

Building the Sample

This sample requires Visual Studio 2012 and the Visual Studio 2012 SDK.  Open the solution and press F5 to run/debug.

Team Explorer Extensibility

Pages and Sections

A single view in Team Explorer is referred to as a page (ITeamExplorerPage).  A page may provide a WPF control that is placed at the top of the client area, but the control is not required.  No two pages can have the same ID in Team Explorer.

A page may contain one or more sections (ITeamExplorerSection) that are arranged in priority order.  A section also provides a WPF control that contains the UI for the section.  Sections may be first-party or third-party extensions to a page.

Home Page - Navigation Items and Links

The Home page is a special Team Explorer page that has its own extensibility.  The Home page contains a list of navigation items (ITeamExplorerNavigationItem) and navigation links (ITeamExplorerNavigationLink).  The list of navigation item plugins is used for both the Home page and the Team Explorer navigation control at the top of the window.

Plugin Interfaces

A plugin should implement the interface for the extension it is providing.  The interfaces are declared in the Microsoft.TeamFoundation.Controls namespace in Microsoft.TeamFoundation.Controls.dll.

Page:
     ITeamExplorerPage

Section:
     ITeamExplorerSection

Navigation Item:
     ITeamExplorerNavigationItem

Navigation Link:
     ITeamExplorerNavigationLink

Plugin Discovery

Plugin discovery for Team Explorer is accomplished using MEF.  The assembly containing any plugins must be installed in one of Visual Studio’s supported extension directories, and must be accompanied by an extension.vsixmanifest file that declares the assembly provides MEF components.  The recommended packaging mechanism is using VSIX and the Visual Studio Extension Gallery.

There is a class attribute for each plugin type that exports the plugin for discovery by Team Explorer.  The attributes are defined in the Microsoft.TeamFoundation.Controls namespace in Microsoft.TeamFoundation.Controls.dll.

Page:
     [TeamExplorerPage("guid")]

Section:
     [TeamExplorerSection("guid", "parentPageGuid", priority)]

Navigation Item:
     [TeamExplorerNavigationItem("guid", priority)]

Navigation Link:
     [TeamExplorerNavigationLink("guid", "parentNavigationItemGuid", priority)]

Plugin Lifetime

No page or section will be instantiated until the user navigates to it for the first time.  The page and all of its sections are then instantiated and then initialized in priority order.  A page or section is only initialized once during its lifetime.  The Initialize method is the appropriate time to query for any services and do any initialization for child controls.  EventArgs provide an IServiceProvider as well as optional initial page context.

In order to manage memory efficiently, only the current page is held in memory at any given time.  When the user navigates to a new page, the current page will have its SaveContext method called in order to preserve any context, and then it is closed.  See the "Context Management" section for more discussion on how that context object is managed.

Context Management

When a user navigates to a page, the caller may provide an optional initial context object.  This object is passed to the page during the Initialize method.  The page can then hydrate its data model and view model asynchronously.

When a page is deactivated, the page/sections SaveContext method will be called before the plugin is disposed.  Each page/section can optionally return a context object that will be preserved by the Team Explorer framework as part of the navigation history.  When the user clicks the back button to navigate back to the previous page, the page/sections will be reconstructed and the same context that was stored during SaveContext will be provided back to them during Initialize.  The page/sections can then use the context to rehydrate their model/viewmodel to the state before the page was closed.

If a user never uses the back button to return to a page, the stored context will eventually fall off the end of the navigation history buffer (which is capped at a reasonable number).  When this happens, the context object will be checked to see if it implements IDisposable, and it will be disposed for you.  If you do not want your context object to be disposed, please wrap it in a container object that does not implement IDisposable.

Sharing Info: Data, Selection, Etc

The page host container implements IServiceContainer/IServiceProvider and provides a place for pages and sections to offer services to each other.  In order to add services to the page host's service container, it may be retrieved by calling GetService(typeof(IServiceContainer)).  GetService calls that cannot be resolved by the page host will be relayed on to the parent IVsPackage.  This allows plugins to discover services in Visual Studio.

The ITeamExplorerPage/Section interfaces also include a GetExtensibilityService method.  This method is called on the page and each section whenever code within Team Explorer calls GetService.  The intent of this design is to allow a page/section to expose its internal data for other sections to consume.  One might just create the extensibility object and place it into the service container, and that would work fine.  What GetExtensibilityService adds is the ability to have the extensibility object lazily created only when it is asked for.  This allows the plugin to avoid spinning up its extensibility objects unless there is a consumer ready to use them.

To better explain this concept, imagine a Team Explorer section plugin is extending the Pending Changes page.  It creates a section class that implements ITeamExplorerSection and chooses the Pending Changes page as its parent.  This will cause the section to be created when the Pending Changes page is created, but how can that section read the current Pending Changes data (included changes, excluded changes, selections, etc)?  The Pending Changes page exposes the IPendingChangesExt extensibility object for just this purpose.  When any section calls GetService(typeof(IPendingChangesExt)), the Pending Changes page will create and return its extensibility object.

If you are building a page or section plugin that you anticipate others will want to extend, then you're encouraged to expose an extensibility object of your own that can be queried.  The suggested naming is IPageNameExt.

Navigation items and links are not given a service provider.  The Visual Studio service provider can be imported using MEF in an ImportingConstructor or as an imported property.  This is an example using an ImportingConstructor:

Navigation

To navigate to a page in Team Explorer, use the NavigateToPage method on on the ITeamExplorer interface (GetService(typeof(ITeamExplorer)).  The method accepts a page ID and an optional intial context object.  The ClosePage method can be used to close the current page, falling back to the previous page in the navigation history.

The current page is exposed via the CurrentPage property.

Notifications

Team Explorer offers a simple notification system where a message can be shown at the top of the Team Explorer window.  A notification consists of a text message, a NotificationType, a NotificationFlags, an optional ICommand callback, and an ID.  Notifications are sent using the ShowNotification method on the ITeamExplorer interface.

The text message for a notification may be simple text, or it may have embedded links using the syntax:

     "Click [here](link_info) to see more information"

When the link is clicked, the ICommand callback will be called, and the link_info text in the parens will be passed as the ICommand param.  If there are multiple links, the callback can differentiate between them by examining the link_info that is passed to the callback.

The NotificationType determines if the notification is an informational, error, or warning message.  Notifications are automatically cleared on any page navigation, but the NotificationFlags.RequiresConfirmation flag may be specified to require the user to acknowledge/dismiss it.

Notifications have a guid for identification.  If a notification is visible and another notification with the same ID is shown, the second will replace the first.  Notifications with different IDs are stacked vertically with a scrollbar.  Notifications can be selectively hidden by calling ITeamExplorer.HideNotification and passing the ID.
